1. Objet du document

Ce cadrage sert de source unique de vérité pour la qualité technique et éditoriale du site LCDMH. Il compile les standards SEO, les règles de publication, les enseignements tirés de cinq audits externes successifs, et les correctifs structurels à appliquer à toute nouvelle page.

Il doit être consulté avant chaque création d'article, chaque génération automatique par LLM, et chaque commit vers GitHub Pages.

2. Règles absolues non négociables

Règle 1 — Encodage UTF-8 sans BOM Ne jamais utiliser PowerShell pour modifier du HTML : problèmes récurrents d'encodage qui cassent les accents. Utiliser Python, VS Code (avec UTF-8 forcé), ou les outils dédiés du projet.
Règle 2 — Aucune invention Ne jamais inventer de données : coordonnées GPS, noms de lieux, prix, références produits, dates. En cas de doute, laisser une formulation neutre (« un lac d'altitude », « aux environs de… ») ou demander confirmation.
Règle 3 — Tout article doit être référencé Un article publié mais non listé dans ARTICLES[] de articles.html est invisible pour Google et les visiteurs. L'ajout dans articles.html fait partie intégrante du pipeline de publication.
Règle 4 — Partenaires centralisés Jamais de lien d'affiliation en dur dans un article. Toujours passer par partenaires.json pour maintenir la cohérence, faciliter les mises à jour et éviter les liens cassés.
Règle 5 — Nom unique par produit Un produit ne doit apparaître que sous un seul nom canonique dans tout le site (title, H1, meta description, OG, texte, Schema.org). Exemple : « Blackview Xplore X1 », et jamais « X1 », « XLOR X1 », « BV8800 Pro » dans le même article.

3. Standards SEO obligatoires

Toute nouvelle page doit respecter la totalité de ces standards avant publication. Ils sont vérifiés lors de chaque audit.

3.1 Balises techniques du <head>

Déclaration de langue sur la balise html :

<html lang="fr">

Meta description entre 120 et 170 caractères, unique, contenant le mot-clé principal et une proposition de valeur :

<meta name="description" content="Test AFERIY Nano 100 : batterie camping moto 99,2 Wh, USB-C 100W, acceptée en cabine avion. Testée sur 5 000 km de road trip en bivouac. Avis terrain honnête.">

Title inférieur à 65 caractères, avec la marque en suffixe :

<title>Test AFERIY Nano 100 : batterie moto 99 Wh USB-C | LCDMH</title>

Canonical URL absolue :

<link rel="canonical" href="https://lcdmh.com/articles/nom-article.html">

Balises OpenGraph complètes : og:title, og:description, og:image, og:url, og:type.

3.2 Structure des titres Hn

  • Un seul H1 par page, contenant le mot-clé principal
  • Pas de deuxième H1 même dans un bloc secondaire — convertir en H2
  • Hiérarchie respectée H1 → H2 → H3 sans saut de niveau

3.3 Images et accessibilité

  • Toute image doit avoir un attribut alt descriptif non vide
  • Pour les miniatures YouTube : alt="Miniature vidéo [sujet] - LCDMH" acceptable
  • Pour les photos : décrire le contenu réel
  • Ajouter loading="lazy" sur les images sous le premier écran (améliore le LCP)

3.4 Maillage interne

  • Minimum 3 liens internes vers d'autres pages pertinentes par article
  • Les pages hub (articles.html, gps.html, pneus.html…) doivent linker vers leurs articles détaillés
  • Ajouter un bloc « Articles liés » en fin de contenu

3.5 Contenu minimum

  • Page hub : minimum 200 mots d'introduction éditoriale
  • Article : minimum 500 mots
  • Page roadbook journalière : minimum 100 mots de récit en plus des données techniques

4. Données structurées Schema.org

Tout article doit inclure un bloc JSON-LD de type Article. Les pages de catégorie utilisent CollectionPage. Les pages principales incluent un BreadcrumbList.

Template Article à copier-coller :

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Titre de l'article",
  "description": "Description de l'article, 120-170 caractères",
  "image": "https://lcdmh.com/images/image-article.jpg",
  "author": {
    "@type": "Person",
    "name": "Yves",
    "url": "https://lcdmh.com/a-propos.html"
  },
  "publisher": {
    "@type": "Organization",
    "name": "LCDMH — Le Coin Des Motards Heureux",
    "logo": {
      "@type": "ImageObject",
      "url": "https://lcdmh.com/images/og-lcdmh.jpg"
    }
  },
  "mainEntityOfPage": {
    "@type": "WebPage",
    "@id": "https://lcdmh.com/articles/nom-article.html"
  },
  "inLanguage": "fr-FR"
}
</script>

5. Templates par type de page

5.1 Page roadbook journalière

Toute page /roadbooks-html/.../jour-XX.html doit contenir en plus des données techniques (distance, temps, coordonnées GPS, prix) :

  • Un paragraphe narratif d'au moins 100 mots (météo, état d'esprit, difficulté, souvenir marquant)
  • Une photo avec alt descriptif
  • Un lien vers la page principale du road trip
  • Un lien vers le jour précédent et le jour suivant
  • Meta description unique adaptée au jour
Note — message "Micro non supporté" Ce message est un fallback JavaScript légitime qui s'affiche uniquement aux navigateurs sans Web Speech API. Ce n'est pas une erreur technique. Recommandation : l'envelopper dans une balise <noscript> ou le masquer par défaut en CSS pour éviter que Googlebot ne l'indexe comme contenu principal de la page.

5.2 Page hub (articles.html, gps.html, pneus.html, photo-video.html…)

Doit contenir une introduction éditoriale de 150 à 300 mots qui explique :

  • le thème de la page
  • le contexte d'expérience (après combien de kilomètres testé)
  • ce que le lecteur va trouver dans la liste qui suit

Ensuite seulement vient le listing ou la grille d'articles.

5.3 Article test produit

Structure recommandée :

  1. Introduction (contexte, qui je suis, pourquoi ce test)
  2. Présentation du produit
  3. Retour terrain après X km
  4. Points positifs
  5. Points négatifs (honnêteté = crédibilité)
  6. Tableau comparatif si pertinent
  7. Verdict
  8. FAQ
  9. Avis communauté
  10. Maillage interne (min. 3 liens)

6. Checklist de validation avant publication

Avant tout push vers GitHub Pages, vérifier ces 10 points sur la nouvelle page :

#Point à vérifier

7. Leçons — fausses alertes récurrentes des audits externes

Cinq audits externes ont été analysés pendant la session de consolidation. Environ 60 % des signalements étaient des fausses alertes. Les auditeurs semblent regarder le HTML rendu après JavaScript dans un navigateur, ce qui masque souvent le <head>. Toujours vérifier dans le code source brut avant de corriger.

Fausse alerte 1 — « Meta-description ABSENTE » Revient à chaque audit. Vérification sur 60 fichiers : toutes les vraies pages ont une meta description valide (120-170 caractères). Seuls les fichiers nav.html et widget-roadtrip-snippet.html n'en ont pas, mais ce sont des snippets inclus et ils ne doivent pas en avoir.
Fausse alerte 2 — « Page 404 contact.html » Vérification directe dans le filesystem : le fichier existe, taille normale, meta description 155 caractères. Cause probable : erreur ponctuelle de cache CDN côté auditeur.
Fausse alerte 3 — « lang="fr" non déclaré » Vérification sur 60 fichiers : seuls 2 snippets n'ont pas la balise, et ils ne doivent pas l'avoir. Toutes les pages principales déclarent bien lang="fr".
Fausse alerte 4 — « Micro non supporté = erreur » C'est une fallback JavaScript légitime pour les navigateurs sans Web Speech API. Ce n'est pas un bug. Voir section 5.1 pour la recommandation d'amélioration.
Fausse alerte 5 — « Vouvoiement incohérent » Le site tutoie systématiquement. C'est la ligne éditoriale assumée, pas une incohérence.
Fausse alerte 6 — « Aucune balise alt sur les images » Vérification : 87 % des images avaient déjà un alt. Les 13 % manquants étaient concentrés sur 3 pages et ont été corrigés en une passe.
Fausse alerte 7 — « Erreurs déjà corrigées » Plusieurs audits rapportent des erreurs déjà corrigées lors d'un audit précédent. Signe que l'auditeur regarde une version cachée ou du cache de son navigateur.

8. Leçons — vraies erreurs récurrentes détectées

Malgré leurs fausses alertes, les audits restent utiles pour détecter ces problèmes réels qui reviennent souvent dans le pipeline de génération automatique :

8.1 Transcriptions orales non relues

Les articles générés à partir des sous-titres YouTube contiennent des erreurs phonétiques sur les noms propres et applications. Exemples corrigés :

  • « Magic Hort » au lieu de Magic Earth
  • « Golard Gels Vegan » au lieu de Gamle Strynefjellsvegen
  • « bibouquer » au lieu de bivouaquer
  • « Biwork » au lieu de bivouac
  • « n'est ce pas de de sater » — bruit non nettoyé

Règle : toute génération par LLM à partir de transcription orale doit passer par une relecture humaine ciblée sur les noms propres (lieux, produits, applications).

8.2 Noms de produits incohérents dans un même article

Un article de test Blackview parlait du produit sous quatre noms différents : X1, XPLORE X1, XLOR X1, BV8800 Pro. Destructeur pour la crédibilité et le SEO. Voir Règle 5.

8.3 Titres et H1 tronqués / copier-coller incomplets

Exemple : « Quel téléphone pour road trip ? Blackview X1 renforcé (pluie, chocs, froid)Blackview X1 :» — cut-off visible. À vérifier systématiquement que le H1 et le <title> se terminent proprement.

8.4 Backticks Markdown résiduels

Les articles générés par LLM peuvent conserver des fences ```html et ``` visibles dans le texte final rendu. À supprimer avant publication.

8.5 Accents manquants dans les titres éditoriaux

Exemple corrigé : « Budget Cap Nord a moto seul : 10 000 km, couts » au lieu de « à moto seul, coûts ». Fréquent quand le titre provient d'un slug URL non accentué.

8.6 Tableaux comparatifs contradictoires avec le texte

Exemple corrigé : article Dunlop Mutant vs Bridgestone T33 — le texte disait « Mutant rincé à 4 800 km », mais le tableau affichait 5 étoiles en longévité au Mutant et 3 au T33. Relire systématiquement texte + tableau ensemble.

8.7 Duplication de contenu

Deux pages racontent le même road trip Alpes 2022 : les-alpes-dans-tous-les-sens.html et alpes-aventure-festival-moto.html. À consolider via un canonical ou à différencier éditorialement.

9. Recommandations d'amélioration priorisées

P1Urgente — impact SEO et stabilité

  • Créer un script validate_seo.py qui parcourt tous les HTML modifiés avant commit et vérifie les 10 points de la checklist. Doit bloquer le push si un point critique manque.
  • Sécuriser publish_to_github() en vérifiant explicitement que les fichiers modifiés sont bien trackés par git avant le push (plusieurs articles sont déjà passés en non-trackés).
  • Corriger fetch_youtube.py si cassé, pour restaurer la mise à jour automatique des données vidéo.

P2Importante — amélioration éditoriale

  • Générateur automatique de meta description à partir du contenu de l'article (premier paragraphe tronqué à 155 caractères, ou génération LLM avec prompt contrôlé et longueur forcée).
  • Générateur automatique d'alt pour les images, basé sur le nom du fichier et le contexte du paragraphe.
  • sitemap.xml auto-généré à chaque push, avec les dates réelles de modification des fichiers.
  • Enrichir les pages roadbook avec un bloc narratif (min. 100 mots) et envelopper le message « Micro non supporté » dans <noscript>.

P3Optimisation — qualité et découvrabilité

  • Consolider la duplication Alpes 2022 (canonical ou différenciation éditoriale)
  • Enrichir les pages hubs (articles.html, gps.html, photo-video.html) avec des introductions éditoriales de 200-300 mots
  • Ajouter un flux RSS pour la découverte des nouveaux articles
  • Uniformiser les noms de produits via un dictionnaire central (partenaires.json étendu)

11. Règles d'infographie et proportions visuelles

Ces règles s'appliquent à toutes les pages qui contiennent des grilles de cartes produit : photo-video.html, gps.html, pneus.html, intercoms.html, equipement.html, hubs partenaires, etc.

Contexte — pourquoi cette règle existe Sur photo-video.html, la règle CSS .card-header { min-height: 65vh; } produisait des vignettes de plus de 700 pixels de hauteur avec une photo de 260 px au milieu, et des emojis 🚁 📸 perdus dans d'immenses zones vides. Résultat visuel catastrophique. Ne jamais reproduire ce motif.

11.1. Hauteur fixe obligatoire pour les vignettes produit

Toutes les vignettes (.card-header ou équivalent) d'une même grille doivent avoir une hauteur fixe et identique. Jamais de min-height exprimé en vh, qui produit des conteneurs de 500 à 700 pixels avec d'énormes vides autour d'une photo ou d'un emoji.

Règle : height: 260px; (valeur recommandée) ou toute valeur en px cohérente dans la grille. Ne pas utiliser min-height sur un conteneur de vignette produit.

.card-header {
  height: 260px;          /* hauteur fixe, jamais vh */
  display: flex;
  align-items: center;
  justify-content: center;
  overflow: hidden;
  position: relative;
}
.card-header img {
  width: 100%;
  height: 100%;
  object-fit: cover;      /* l'image remplit le cadre sans déformation */
  display: block;
}

11.2. Variante emoji-only pour les cartes sans visuel réel

Quand une carte produit n'a pas encore de photo réelle et affiche uniquement un emoji (🚁 📸 🏔️ 🔦), utiliser une classe supplémentaire emoji-only pour :

  1. afficher un fond plus contrasté (gradient orange clair)
  2. agrandir l'emoji (font-size: 8rem) pour qu'il remplisse visuellement l'espace disponible
  3. ajouter un badge « Visuel à venir » en bas de la carte, pour signaler qu'il s'agit d'un placeholder et non d'un choix design volontaire
.card-header.emoji-only {
  background: linear-gradient(135deg, #fff4e6 0%, #ffe0b2 100%);
  font-size: 8rem;
}
.card-header.emoji-only::after {
  content: "Visuel à venir";
  position: absolute;
  bottom: 12px; left: 0; right: 0;
  text-align: center;
  font-size: .72rem;
  color: #b37400;
  letter-spacing: .05em;
  text-transform: uppercase;
  font-weight: 600;
}

En HTML, appliquer simplement la classe complémentaire : <div class="card-header emoji-only">🚁</div>.

11.3. Alignement vertical de tous les headers d'une même grille

Dans une grille CSS, toutes les cartes d'une même ligne s'étirent automatiquement à la hauteur de la carte la plus haute. Cela provoque des décalages visibles si les .card-body ont des longueurs de texte très différentes. Pour garder une grille propre et alignée :

  • hauteur du header : height: 260px; fixe et identique pour toutes les cartes de la grille
  • corps de carte : utiliser display: flex; flex-direction: column; sur .camera-card (ou équivalent) pour que le bouton en bas reste aligné même si le texte est plus court
  • bouton d'action : ajouter margin-top: auto; pour le coller au bas de la carte, ce qui garantit l'alignement horizontal des boutons dans chaque ligne de la grille

11.4. Interdiction des grands vides autour des images

Tout cadre qui contient une image produit doit obéir à l'une de ces trois règles :

  1. l'image remplit tout le cadre avec object-fit: cover, aucun vide visible
  2. l'image est centrée avec un padding uniforme d'au maximum 20 % de la hauteur totale du cadre
  3. le cadre est d'une taille proche de la taille native de l'image (±10 %)

Une vignette qui montre 200 pixels d'image au milieu de 500 pixels de vide est interdite. C'est systématiquement une erreur de CSS (min-height: Nvh en général) qui doit être corrigée avant publication.

11.5. Checklist infographie avant publication

Avant tout push d'une page contenant une grille de cartes produit, vérifier ces cinq points :

  1. ouvrir la page dans le navigateur et passer en mode responsive sur trois largeurs : 375 px (mobile), 768 px (tablette), 1280 px (desktop)
  2. vérifier que toutes les cartes d'une même ligne ont la même hauteur sur les trois largeurs
  3. vérifier qu'aucune image n'est entourée de plus de 40 pixels de vide au-dessus ou en-dessous
  4. vérifier que les cartes emoji-only ont bien le fond contrasté orange + badge « Visuel à venir »
  5. vérifier que les boutons d'action (Lire le test, Acheter, Voir le test…) sont alignés sur la même ligne horizontale dans chaque ligne de la grille
Règle mnémonique Hauteur fixe en pixels, jamais en vh. object-fit: cover pour les photos. Classe emoji-only pour les placeholders. Boutons alignés en bas avec margin-top: auto;.
